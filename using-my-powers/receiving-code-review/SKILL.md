# Receiving Code Review — 接收代码审查

代码审查需要技术评估，而非情感表演。

**宣告：** 「我正在使用 my-powers:receiving-code-review」

**核心原则：** 验证后再实施。不清楚就先问。技术正确性优先于社交舒适感。

**违反本规则的字面内容，就是违反本规则的精神。**

## 响应模式

```
收到代码审查反馈时：

1. 阅读：完整阅读所有反馈，不急于响应
2. 理解：用自己的话复述需求（或提问澄清）
3. 验证：对照代码库核实
4. 评估：对当前代码库技术上是否合理？
5. 响应：技术性确认或有理由的推回
6. 实施：逐条处理，每条单独测试
```

## 禁止的响应

**绝不说：**
- "你说得完全正确！"（表演性认同）
- "好观点！" / "感谢指出！"（空洞的赞美）
- "让我马上实现"（验证之前）
- 任何形式的感谢表达

**而应该：**
- 复述技术需求
- 提出澄清问题
- 用技术理由推回（如果建议有问题）
- 直接动手（行动优于言辞）

## 处理不清晰的反馈

```
如果有任何条目不清楚：
  停下来——不要先实施任何条目
  对不清楚的条目提问澄清

原因：各条目可能相互关联。理解不完整 = 实施错误。
```

**示例：**
```
反馈："修复第 1-6 条"
理解了 1、2、3、6，不理解 4、5。

❌ 错误：先实施 1、2、3、6，之后再问 4、5
✅ 正确："我理解第 1、2、3、6 条，请在我开始之前说明第 4、5 条。"
```

## 按反馈来源处理

### 来自用户（项目所有者）
- **可信** — 理解后实施
- **仍需澄清** — 范围不清时先问
- **不要表演性认同**
- **直接行动** 或技术性确认

### 来自外部 Reviewer

```
实施之前：
  1. 检查：对当前代码库技术上是否正确？
  2. 检查：是否会破坏现有功能？
  3. 检查：当前实现是否有其存在的原因？
  4. 检查：是否在所有平台/版本上都有效？
  5. 检查：reviewer 是否了解完整的上下文？

如果建议看起来有问题：
  用技术理由推回

如果无法轻易验证：
  直接说明："我无法在不 [X] 的情况下验证这一点。是否应该 [调查/询问/继续]？"

如果与用户之前的决策冲突：
  停下来，先与用户讨论
```

## YAGNI 检查

```
如果 reviewer 建议"正确实现"某个功能：
  grep 代码库，查看实际使用情况

  如果未使用："这个接口没有调用方。删除它（YAGNI）？"
  如果有使用：则正确实现
```

## 实施顺序

```
对于多条反馈：
  1. 先澄清所有不清楚的条目
  2. 按以下顺序实施：
     - 阻塞性问题（破坏性、安全性）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 每条单独测试
  4. 确认没有引入回归
```

## 何时推回

以下情况推回：
- 建议会破坏现有功能
- Reviewer 缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对当前技术栈技术上不正确
- 存在历史遗留或兼容性原因
- 与用户的架构决策冲突

**如何推回：**
- 使用技术理由，不要情绪化
- 提出具体问题
- 引用可用的测试或代码
- 如涉及架构，请用户介入

## 确认正确的反馈

当反馈确实正确时：
```
✅ "已修复。[简短说明改了什么]"
✅ "发现得好——[具体问题]。已在 [位置] 修复。"
✅ [直接修复，在代码里体现]

❌ "你说得完全正确！"
❌ "好观点！"
❌ "感谢指出！"
❌ 任何形式的感谢
```

## 优雅地纠正自己的推回

如果推回后发现自己错了：
```
✅ "你是对的——我检查了 [X]，它确实 [Y]。现在实施。"
✅ "已验证，你是对的。我最初的理解有误，原因是 [reason]。正在修复。"

❌ 长篇道歉
❌ 为当初的推回辩解
❌ 过度解释
```

陈述纠正，继续前进。

## 常见错误

| 错误 | 修正 |
|------|------|
| 表演性认同 | 陈述需求或直接行动 |
| 盲目实施 | 先对照代码库验证 |
| 批量处理但不测试 | 逐条实施，每条测试 |
| 默认 reviewer 正确 | 检查是否会破坏现有功能 |
| 回避推回 | 技术正确性优先于舒适感 |
| 部分实施 | 先澄清所有条目 |
| 无法验证却继续 | 说明局限，请示方向 |

## GitHub 评论回复

在 GitHub 内联评论线程中回复时，回复到对应评论线程（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），而不是作为顶层 PR 评论。

## 底线

**外部反馈 = 需要评估的建议，而非需要执行的命令。**

验证。质疑。然后实施。

不要表演性认同。技术严谨永远优先。
