# Spec Split Checker 子智能体 Prompt 模板

当所有 splitter 子智能体完成后，为每个板块并行派发一个检查子智能体时，使用此模板。

**用途：** 验证拆分结果相对于原始 spec 的迭代计划，功能覆盖不交且完全

**派发时机：** `splitting-specs` 第 3 步，所有 splitter 完成后，每个板块对应一个检查子智能体，并行执行

```
Agent（general-purpose）:
  description: "检查 [<module-name>] 迭代计划拆分的完全性与不交性"
  prompt: |
    你的任务是验证板块「<module-name>」的迭代计划拆分结果满足两个性质：
    完全性（原始 spec 中的每条功能恰好出现在某一阶段文件中）
    与不交性（任意两个阶段文件的实现范围不包含相同条目）。

    **输入文件（直接读取）：**
    - 原始 spec：`docs/my-powers-output/specs/YYYY-MM-DD-<module>-spec.md`
    - Core 阶段文件：`docs/my-powers-output/specs/YYYY-MM-DD-<module>-spec-core.md`
    - Update 阶段文件（由协调者列出）：
      - `docs/my-powers-output/specs/YYYY-MM-DD-<module>-spec-update-1-<feature>.md`（如有）
      - `docs/my-powers-output/specs/YYYY-MM-DD-<module>-spec-update-N-<feature>.md`（如有）

    **不修改任何文件。**

    ## 检查流程

    **第 1 步：提取原始 spec 的完整功能集 F**

    读取原始 spec「迭代计划」章节，逐阶段提取所有功能条目，构建完整功能集 F。
    每条功能记录为：
    - **名称**：spec 中的功能名称原文（不改写）
    - **摘要**：对该功能内容的一句话概括（用自己的语言简明表达，不照抄正文）

    **第 2 步：提取各阶段文件的实现范围**

    读取每个阶段文件的「实现范围」（或「Core 实现范围」/「本阶段功能范围」）章节，
    提取各自声明实现的功能条目，同样记录名称（原文）与摘要，构成覆盖集 S_core, S_update_1, …

    注意：阶段文件中的「暂不实现」列表是排除声明，不计入该阶段的覆盖范围。

    后续检查以**名称**作为匹配键；摘要用于在上报时帮助读者快速理解问题所在。

    **第 3 步：完全性检查**

    验证：F 中的每条功能都出现在某个 S 中。
    找出 F 中未被任何阶段文件覆盖的条目（遗漏功能）。

    **第 4 步：不交性检查**

    对所有阶段文件两两比较各自的覆盖集，找出同时出现在多个阶段文件「实现范围」中的条目（重复功能）。

    ## 上报格式

    （每个问题条目的问题类型由其所在章节标题声明：完全性问题 / 不交性问题）

    ```
    状态：PASS | FAIL

    完全性问题（如有，每条单独列出）：
    - 「<功能名称原文>」（<一句话摘要>）未出现在任何阶段文件的实现范围中

    不交性问题（如有，每条单独列出）：
    - 「<功能名称原文>」（<一句话摘要>）同时出现在 <文件 A> 和 <文件 B> 的实现范围中
    ```

    所有问题均 PASS 时输出 `状态：PASS`，无需列出其他内容。
```

**检查者返回：** PASS，或 FAIL + 具体问题列表

**后续处理：**
- PASS → 该板块通过，等待所有板块完成
- FAIL → 协调者根据问题修复对应阶段文件 → 重新派发该板块的检查子智能体，直到 PASS
