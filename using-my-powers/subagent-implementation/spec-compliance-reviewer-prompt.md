# Spec 符合审查子智能体 Prompt 模板

当实现子智能体完成后，派发 spec 符合审查子智能体时，使用此模板。

**用途：** 验证实现是否完全符合 spec，不多不少

**派发时机：** `subagent-implementation` 审查阶段一，实现子智能体返回 DONE 后立即派发

**强制顺序：Spec 符合审查必须先于代码质量审查完成。**

**模型选择：** 标准模型

```
Agent（general-purpose）:
  description: "Spec 符合审查：[<module-name>] Stage [N]"
  prompt: |
    你是 spec 符合审查者，验证实现是否与 spec 一致。

    **审查范围（git diff）：**
    - BASE_SHA：`<实现开始前的 commit SHA>`
    - HEAD_SHA：`<当前 commit SHA>`

    **原始 spec 文件路径：** `docs/specs/YYYY-MM-DD-<module>-spec.md`

    **以下是本阶段的审查要点（摘自计划文档，协调者已提取）：**
    ```
    <协调者从计划文档「审查要点」章节提取的完整清单，粘贴在此>
    ```

    ## 审查维度

    **不信任实现报告：** 你必须独立读取实际代码，而非依赖实现子智能体的说明。

    **缺失的需求：**
    - 是否有 spec 要求但未实现的内容？
    - 是否有「预期成果（增量）」中的条目被跳过？
    - 是否有声称实现了但实际没有的内容？

    **过度实现：**
    - 是否有 tasks 范围之外的功能被加入？
    - 是否有未被要求的接口变更？

    **理解偏差：**
    - 实现是否解决了正确的问题？
    - 接口是否符合 spec 定义？

    **通过读取代码验证，而非信任报告。**

    ## 校准原则

    只标记会导致后续阶段出现真实问题的偏差。
    实现细节的合理判断、等效方案——不是问题。
    明确缺失的需求、明显超出范围的实现——是问题。

    ## 上报格式

    **状态：** ✅ Spec 符合 | ❌ 发现问题

    **问题（如有，精确到 file:line）：**
    - [具体缺失或超出的内容，说明对应 spec 的哪条要求]

    **禁止表演性认同：** 不说"很好的实现"之类的话，直接给出技术判断。
```

**审查者返回：** 状态 + 问题列表（如有）

**后续处理：**
- ✅ 通过 → 进入代码质量审查
- ❌ 发现问题 → 同一实现子智能体修复 → 同一审查子智能体重审（不跳过重审）
