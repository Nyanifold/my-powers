# Subagent Implementation — 子智能体驱动实现

按阶段依赖顺序调度实现子智能体，每个任务阶段完成后经过两阶段审查，通过后写实现报告。

**宣告：** 「我正在使用 my-powers:subagent-implementation」

**核心原则：**
- 每个任务阶段对应一个独立的实现子智能体，上下文精确构造，不继承会话历史
- 两阶段审查强制顺序：先 spec 符合审查，再代码质量审查
- 验证先于声明：没有运行过验证命令，不得宣称任何阶段完成
- 连续执行：除非遇到无法解决的 BLOCKED，不中途询问用户是否继续

## 第 0 步：分支隔离检查

**在任何实现开始之前：**

```bash
BRANCH=$(git branch --show-current)
```

- 如果当前在 `main` 或 `master`：必须创建新分支，不得在主分支上直接实现
- 如果已在功能分支：继续
- 询问用户偏好，或默认创建 `feature/<system-name>-<date>` 分支

## 第 1 步：提取所有任务，构建执行队列

一次性读取所有计划文件，提取每个任务的关键信息，用 TodoWrite 建立完整任务列表：

```
[板块名] Stage N — <阶段名称>
  计划文件: docs/my-powers-output/plans/YYYY-MM-DD-<module>-task-N-plan.md
  前置依赖: [列表]
  状态: 待执行
```

按依赖关系构建执行顺序：无依赖的阶段最先执行，有跨板块依赖的阶段等待其前置完成后执行。

## 第 2 步：执行循环（每个任务阶段）

### 派发实现子智能体

**实现子智能体获得的精确上下文（协调者构造，不让子智能体自行搜索）：**
- 板块名称、阶段编号、阶段名称
- 对应 spec 文件路径（协调者提供，子智能体自行读取内容）
- 对应 tasks 文件中该阶段的完整文本（协调者提取后传入）
- 计划文档的完整路径（协调者提供，子智能体自行读取内容）
- 当前 git 分支名
- 「实现子智能体职责」章节的完整文本

**模型选择：**
- 计划明确、仅涉及 1-2 个文件：低成本模型
- 多文件集成、需要判断的任务：标准模型
- 跨板块接口、架构判断：最强模型

**先答疑再实现：**
实现子智能体上报 `NEEDS_CONTEXT` 时，协调者回答问题后重新派发，不催促子智能体在信息不足时继续。

### 处理子智能体状态上报

实现子智能体必须返回以下四种状态之一：

**`DONE`：** 正常完成，进入 spec 符合审查。

**`DONE_WITH_CONCERNS`：** 完成但有疑虑。协调者先阅读疑虑内容：
- 疑虑涉及正确性或范围 → 处理后再审查
- 疑虑是观察性说明（如"这个文件变大了"）→ 记录后继续审查

**`NEEDS_CONTEXT`：** 需要更多信息。协调者提供缺失上下文后重新派发同一任务。

**`BLOCKED`：** 无法继续。协调者评估：
1. 上下文问题 → 补充上下文，重新派发
2. 需要更强推理能力 → 升级模型，重新派发
3. 任务过大 → 拆分后重新派发
4. 计划本身有问题 → 升级给用户决定

**绝不**忽略 BLOCKED 或用相同配置强行重试。

### 协调者自审（审查前）

收到实现子智能体的 DONE 报告后，在派发任何审查子智能体前，协调者快速检查：

- [ ] 实现子智能体已完成自检（第 5 步）且状态为 DONE
- [ ] git diff 范围与该阶段预期成果一致（无明显遗漏，无明显范围外实现）
- [ ] 新增文件/函数命名与 spec 约定一致

发现明显问题直接反馈给实现子智能体修复，再继续派发审查。

### 审查阶段一：Spec 符合审查

**强制顺序：Spec 符合审查必须先于代码质量审查完成。**

获取 git SHA：
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或该任务开始前的 commit
HEAD_SHA=$(git rev-parse HEAD)
```

派发 Spec 符合审查子智能体（标准模型），提供：
- 原始 spec 文件路径
- tasks 文件中该阶段的「审查要点」清单
- `BASE_SHA` 和 `HEAD_SHA`（用于定位 diff 范围）

审查维度：
- 所有「预期成果（增量）」是否都已实现？
- 是否有范围外的实现（过度实现）？
- 接口是否符合 spec 定义？

审查结果：
- ✅ 通过 → 进入代码质量审查
- ❌ 发现问题 → 同一实现子智能体修复 → 同一审查子智能体重审（不跳过重审）

**禁止表演性认同：** 审查子智能体发现问题后，用技术语言描述，不说"很好的发现"之类的话，直接列出问题清单。

### 审查阶段二：代码质量审查

派发代码质量审查子智能体（最强模型），提供：
- 同样的 `BASE_SHA` 和 `HEAD_SHA`
- 板块的技术栈（来自 spec）

审查维度：
- 代码结构是否清晰？文件职责是否单一？
- 测试是否覆盖了计划中的测试点？
- 是否有明显的代码质量问题（重复代码、魔法数字、未处理的错误路径）？

审查结果：
- ✅ 通过 → 写实现报告，标记任务完成
- ❌ 发现问题 → 同一实现子智能体修复 → 同一审查子智能体重审

**禁止表演性认同：** 同上。

### 写实现报告

两阶段审查均通过后，实现子智能体（或协调者）写实现报告：

文件路径：`docs/my-powers-output/reports/YYYY-MM-DD-<module>-task-<N>-report.md`

```markdown

> 生成日期：YYYY-MM-DD
> 对应计划：docs/my-powers-output/plans/YYYY-MM-DD-<module>-task-<N>-plan.md

```

报告写完提交 git，然后在 TodoWrite 中标记该任务完成，继续下一个任务。

## 第 3 步：全局最终审查

### 协调者自审（全局审查前）

所有任务阶段完成后，在派发最终审查子智能体前，协调者检查：

- [ ] 所有任务阶段均已标记完成（无 BLOCKED/PARTIAL 遗留）
- [ ] 各板块接口实现与 modules.md 中的定义无明显出入
- [ ] 演示脚本/命令已全部就绪，可实际运行

发现问题直接修复，再派发全局审查子智能体。

所有任务阶段完成后，派发最终代码审查子智能体（最强模型），对整个实现进行全局审查：

**全局审查维度：**
- 各板块接口是否与 spec 定义一致？
- 板块间的依赖关系是否符合 modules 文件中的约定？
- 是否有跨阶段引入的一致性问题？
- 所有演示是否可运行？

审查通过后，调用 finishing-a-module 子技能 — 读取 `finishing-a-module/SKILL.md` 技能文件

## 连续执行规范

**停止并升级给用户的唯一理由：**
- BLOCKED 状态且评估后无法在当前上下文中解决
- 计划文档本身有错误或矛盾，需要用户决策

**不停下来询问的情况：**
- 常规任务完成，继续下一个
- 审查发现问题，修复后继续
- 中间结果汇报（直接继续，不等用户回复）

## 实现子智能体职责

> 以下内容由协调者完整复制进实现子智能体的 prompt。

你是一个实现子智能体，负责完成指定任务阶段的实现工作。你不继承任何会话历史，所有上下文由协调者提供。

**你的任务：**
板块「<module-name>」Stage <N>：<阶段名称>

**输入文件路径（读取这些文件获取详情）：**
- spec：`docs/my-powers-output/specs/YYYY-MM-DD-<module>-spec.md`
- 计划：`docs/my-powers-output/plans/YYYY-MM-DD-<module>-task-<N>-plan.md`

**以下是本阶段的任务说明（摘自 tasks 文件，协调者已提取）：**
```
<任务说明文本>
```

**实现规范：**
1. 按计划文档的步骤顺序实现
2. **TDD 铁律**：每个功能点必须先写失败测试，确认测试以正确原因失败，再写最少代码令其通过，再确认没有破坏其他测试
3. 实现完成后运行完整测试套件，确认通过
4. 搭建并验证演示能正常运行
5. 自检：对照计划文档的「审查要点」逐条确认
6. 提交所有变更（分多次提交，每个逻辑单元一次提交）

**TDD Red Flags（遇到以下情况立即停止）：**
- 先写了实现代码，再补测试
- 测试一写出来就通过了（说明没有测到正确的东西）
- 跳过"确认测试失败"这一步
- "这个很简单，不需要测试"

**遇到 bug 时：** 遵循系统化调试流程——先确认复现、再找根因、再修复、再验证。修复通过后在 `docs/my-powers-output/reports/` 写 bugfix 记录：`YYYY-MM-DD-<module>-task-<N>-bugfix.md`（同一 stage 多个 bug 则加序号：`-bugfix-1.md`、`-bugfix-2.md`）。

**验证先于声明：** 没有运行过测试命令，不得说"测试通过"；没有运行过演示命令，不得说"演示可用"。

**完成后上报状态（必须是以下四种之一）：**
- `DONE`：所有步骤完成，测试通过，演示可运行
- `DONE_WITH_CONCERNS`：完成但有疑虑，说明具体疑虑内容
- `NEEDS_CONTEXT`：需要某些信息才能继续，说明需要什么
- `BLOCKED`：遇到无法自行解决的问题，详细说明阻塞原因

## Red Flags

**绝不：**
- 在 main/master 分支上直接开始实现（第 0 步检查）
- 跳过 Spec 符合审查直接进行代码质量审查（顺序强制）
- 审查发现问题后跳过重审（发现问题 = 未完成）
- 在没有运行验证命令的情况下宣称测试通过或阶段完成
- 忽略 BLOCKED 状态或用相同配置强行重试
- 中途询问用户"是否继续"（除非 BLOCKED 需要升级）
- 让实现子智能体继承协调者的会话上下文
- 在 Spec 符合审查未通过时进入代码质量审查
- 先写实现代码，再补测试（TDD 铁律违反）
- 跳过确认测试失败的步骤
- 修复 bug 后不写 bugfix 记录
