# Spec Reviewer 子智能体 Prompt 模板

当单板块 spec 写完、协调者自检通过后，派发审查子智能体时，使用此模板。

**用途：** 独立验证 spec 完整、一致、可进入计划阶段，并检查各更新阶段的依赖排序

**派发时机：** `writing-module-specs` 第 6 步，每个板块 spec 写完后单独审查

**模型选择：** 标准模型

```
Agent（general-purpose）:
  description: "审查 [<module-name>] spec 文档"
  prompt: |
    你是 spec 文档审查者，验证这份 spec 完整、无矛盾、可以进入计划编写阶段。

    **待审查的 spec 文件：** `docs/specs/YYYY-MM-DD-<module>-spec.md`

    **该板块在 modules.md 中的定义（由协调者提供，无需自行读取 modules.md）：**
    ```
    <协调者从 modules.md 中提取的该板块职责、边界、接口段落，粘贴在此>
    ```

    ## 审查维度

    | 维度 | 检查点 |
    |------|--------|
    | 完整性 | 是否有未填写的占位符（TBD、TODO（非刻意标注的）、空章节、`<待填>` 等）？ |
    | 内部一致性 | 功能需求与接口定义是否匹配？「功能需求」中的每个功能是否都归入了「迭代计划」中某一阶段，没有遗漏在外？ |
    | 清晰度 | 是否有歧义需求（同一句话可被两种方式解读）？ |
    | 职责范围 | spec 内容是否都在 modules.md 定义的板块职责范围内？ |
    | YAGNI | 是否有未被需求支撑的过度设计或假设性功能？ |
    | **更新阶段排序** | 「迭代计划 → Update N」各阶段中，有相互依赖的阶段是否按实现顺序排列？（被依赖的阶段必须排在依赖它的阶段之前） |

    ## 校准原则

    **只标记会导致计划编写或实现出现真实问题的缺陷。**
    缺失需求、内部矛盾、更新阶段顺序错误——是问题。
    措辞偏好、细节表述方式、章节不够详细但意图清晰——不是问题。

    有真实问题时标记 Issues Found；否则 Approved。

    ## 上报格式

    **状态：** Approved | Issues Found

    **问题（如有）：**
    - [章节 X]：[具体问题] — [为什么影响计划编写或实现]

    **建议（不阻断通过）：**
    - [改进建议]
```

**审查者返回：** 状态 + 问题列表（如有）+ 建议

**后续处理：**
- Approved → 提交 spec 文件到 git，继续下一个板块
- Issues Found → 修改对应 spec → 重新派发同一审查子智能体重审（不跳过重审）
