# Writing Module Specs — 板块规格编写

对每个叶子板块依次进行头脑风暴，编写完整的规格文档。所有 spec 生成完毕后一并交用户审阅，用户确认后进入 splitting-specs。

**宣告：** 「我正在使用 my-powers:writing-module-specs，当前板块：<name>」

**任务跟踪：** 对每个叶子板块，使用 TaskCreate 创建一个「板块 <name> spec」任务，开始时标记为 in_progress，审查通过并提交后标记为 completed。

<HARD-GATE>
用户未确认所有 spec 之前，不得调用 splitting-specs。每个板块的 spec 必须包含「迭代计划」章节。

spec 必须明确所有技术选型与具体算法——不得遗留任何「待确定」。唯一例外：用户在回答问题时显式说明「这是后续版本的工作」。用户确认 spec 之后，不再参与这些决策。
</HARD-GATE>

## Anti-Pattern：「这个板块很简单，不需要逐步确认」

每个板块都走这个流程。省略「初稿确认」只会把分歧推到实现阶段，届时返工成本高得多。spec 可以很短，但必须通过讨论写成、经用户确认，不能自行假设需求细节。

**「用户肯定知道我在做什么」不等于「用户确认了这份 spec」。**

## 处理顺序

1. 读取 `docs/my-powers-output/YYYY-MM-DD-modules.md`，提取所有叶子板块列表
2. 按依赖顺序排列叶子板块（被依赖的板块先做）
3. 串行对每个板块执行「单板块头脑风暴流程」
4. 所有板块完成后，列出所有 spec 文件，请用户一并审阅
5. 用户确认（或标注 TODO）后，调用 splitting-specs

## 单板块头脑风暴流程

### 第 1 步：宣告与上下文回顾

宣告当前板块，并简要复述来自板块说明文档的定义：
> 「开始对板块「<name>」进行 spec 头脑风暴。根据板块说明文档，该板块的职责是 <...>，对外接口包括 <...>。」

**关于 modules.md 中的内部细节：**

boards.md 的重点是边界与接口，但有时会附带提到板块内部的一些实现方向（如数据结构名称、存储方式、内部分层等）。这些内容**不构成约束**——spec 阶段从零开始讨论内部设计，重新得出的结论只要不与 modules.md 中定义的**边界与对外接口**冲突，都是合法的。不要把 modules.md 里的内部细节当作已定结论跳过讨论。

### 第 2 步：逐一提问澄清需求

**规则：**
- 每次只问一个问题，等待用户回答后再问下一个
- 每次问题优先提供 3-5 个可选项，并为每个选项简要说明核心取舍，帮助用户做出有依据的选择（而不只是列出名称）
- 需要展示结构或关系时，可使用 ASCII 树、ASCII 框图或 mermaid/dot 语法辅助说明

**覆盖以下维度（根据板块性质选择适用的）：**

- **功能需求细化**
  - 核心功能的具体行为是什么？
  - 是否有优先级区分？
  
- **约束与非功能需求**
  - 性能要求？延迟上限？吞吐量？
  - 可靠性要求？是否需要重试、幂等？
  - 安全边界？哪些数据需要保护？

- **接口细节**
  - 数据结构的具体字段？
  - 错误情况如何分类和返回？
  - 是否有版本兼容性考虑？

- **与相邻板块的边界情况**
  - 依赖的板块返回错误时，本板块如何处理？
  - 是否有状态需要跨板块同步？

- **技术选型与算法（必须在本轮讨论中全部确定）**
  - 核心功能采用哪种算法？（例：搜索用倒排索引还是向量检索；调度用优先队列还是时间轮）
  - 采用哪个库或框架？
  - 存储方案？（关系型 / 文档型 / KV 的具体选择）
  - 任何其他影响实现方式的选型？
  - **规则：用户未显式说明「后续版本」的选型，必须在此轮讨论中完全确定，不留待后续。**

- **迭代计划**
  - 哪些功能是系统正常运行的最小集合（Core）？
  - 是否存在自然的功能分组，适合分多个更新阶段交付？
  - 各更新阶段之间是否有依赖顺序？
  - 根据功能规模与依赖关系，建议几个更新阶段？

### 第 3 步：呈现设计方案

分段呈现，每段确认后再继续：
1. **架构概述** — 板块内部的主要组成部分（复杂时附类图）
2. **接口细化** — 比板块说明文档更详细的接口定义（持久化板块附 ER 图）
3. **错误处理策略** — 各类错误的处理方式
4. **迭代计划** — Core 的最小集合范围，以及各更新阶段的功能边界与顺序

### 第 4 步：初稿确认

呈现所有设计方案后，**在写正式 spec 文件之前**，执行以下步骤：

1. 将讨论结论写入 `docs/my-powers-output/drafts/YYYY-MM-DD-<module>-spec-draft.md`，用 `- [ ]` 列出每个要点：

```markdown
## 要点确认清单

- [ ] **架构概述**：<内部组成部分>
- [ ] **对外接口**：<接口定义摘要>
- [ ] **依赖的外部接口**：<依赖摘要>
- [ ] **错误处理策略**：<策略>
- [ ] **迭代计划 Core**：<功能列表>
- [ ] **迭代计划 Update N**：<功能列表>（如有）
```

2. 逐条询问用户（每次只问一条）。对每条要点，需**完整展开该条目的内容**，并**说明这样决定的原因**——为什么在讨论的选项中选择了这个设计。示例格式：

   > 「请确认**架构概述**：
   >
   > [内部组成部分及协作方式的完整描述]
   >
   > **原因说明：** [为什么这样设计——例如将 X 从 Y 中分离出来是为了让热路径不受 Y 的 I/O 开销影响；使用单一 Z 组件是为了避免需要共识协议的分布式状态。]
   >
   > 是否正确？」

3. 用户确认后改为 `- [x]`；有修改时更新初稿并重新确认。

4. 所有条目 `- [x]` 后，向用户确认：
   > 「所有要点已确认。是否可以开始撰写正式 spec 文件？」
   等待用户明确同意后，继续第 5 步。

### 第 5 步：编写 spec 文件

文件路径：`docs/my-powers-output/specs/YYYY-MM-DD-<module>-spec.md`

```markdown
# <板块名> Spec

> 生成日期：YYYY-MM-DD
> 所属系统板块说明：docs/my-powers-output/YYYY-MM-DD-modules.md

## 概述

<2-3 句话说明该板块的定位与作用>

## 职责与边界

**职责：**
<列表>

**不包含（边界）：**
<列表>

## 架构概述

<说明板块内部的主要组成部分及其协作方式>

<!-- 图表说明：
  - 内部结构简单（≤3 个类/组件）：文字描述或 ASCII 框图即可
  - 内部有多个类/结构体协作：附类图（UML Class Diagram），
    可用 mermaid/dot classDiagram 语法，展示类的属性、方法、继承与组合关系
  - 有复杂的并发流程或业务分支：附活动图（Activity Diagram），
    可用 mermaid/dot flowchart 语法
  - 板块是数据处理/ETL 类型：附数据流图（DFD），
    可用 mermaid/dot flowchart 或 ASCII 框图展示数据如何在内部流动和变换
-->

## 接口定义

### 对外暴露的接口

<每个接口一节，包括：>
- 名称与协议（HTTP/gRPC/消息队列/函数调用等）
- 请求/输入参数（字段名、类型、是否必填、说明）
- 响应/输出（字段名、类型、说明）
- 错误码与含义

<!-- 图表说明（持久化板块）：
  如果该板块管理持久化数据，在此附 ER 图，
  展示实体、字段类型、主键/外键、关系（一对多/多对多等）。
  可用 mermaid/dot erDiagram 语法。
  ER 图是接口定义的一部分，不是可选附录。
-->

### 依赖的外部接口

<本板块调用其他板块的接口，同样格式>

## 功能需求

<每个功能一节，描述预期行为，包括正常路径和异常路径>

<!-- 图表说明：
  - 功能涉及明确的状态转换（订单状态、连接状态、任务生命周期等）：
    附状态机图（State Machine Diagram），展示状态、触发事件、转换条件。
    可用 mermaid/dot stateDiagram-v2 语法。
  - 判断标准：如果一个对象有超过 3 种状态且转换条件不同，就值得画状态机图
-->

## 非功能需求

- **性能：** <延迟/吞吐量要求，或"暂无硬性要求">
- **可靠性：** <重试策略、幂等性要求>
- **安全：** <认证、授权、数据保护要求>

## 迭代计划

> 本节定义本板块的迭代阶段。阶段数量由功能规模与依赖关系决定，可以是一个到多个。
> 每个阶段在前一阶段基础上叠加，不破坏已有接口。

### Core：<最小可运行集名称>

> Core 完成后，板块应能独立运行，系统可以正常工作。

- <功能 1>：<简要说明>
- <功能 2>：<简要说明>

### Update 1：<feature-name>

> 在 Core 基础上叠加，不破坏现有接口。

- <功能 A>：<简要说明，包括与前序阶段的接口差异（如有）>

<!-- 如有更多更新阶段，按如下格式继续添加。如无，删除此注释和下方示例。 -->

### Update N：<feature-name>

- <功能 X>：<简要说明>

## 开放问题（TODO）

<仅限用户在讨论中显式说明「这是后续版本的工作」的内容。技术选型、算法选择、接口字段、错误处理方式等实现所需的所有决策不得出现在此节——这些必须在 spec 中完全确定。>
```

## 图表使用规范（spec 范围）

| 图表类型 | 放置章节 | 使用条件 |
|----------|----------|----------|
| 类图（UML Class Diagram） | 架构概述 | 板块内部有 ≥ 4 个类/结构体协作 |
| 活动图（Activity Diagram） | 架构概述 | 有复杂并发分支或多路径业务流程 |
| 数据流图（DFD） | 架构概述 | 数据处理/ETL 类板块，强调数据变换而非控制流 |
| ER 图 | 接口定义 | 板块管理持久化数据（必须附，不可选） |
| 状态机图（State Machine） | 功能需求 | 存在 ≥ 3 种状态且转换条件不同的对象 |

**不在 spec 里放的图：** 时序图（跨板块交互）、组件图、部署图——这些属于 modules.md。

### 第 6 步：自审

写完 spec 文件后，逐条检查以下要点，发现问题直接修改，无需记录：

- [ ] Iteration Plan 节存在，Core 范围已明确（无"待定"占位）
- [ ] 每个功能需求都覆盖了 happy path 和 error path
- [ ] 接口定义到字段级别（字段名/类型/是否必填），非仅协议名称
- [ ] 边界（不包含）一节非空
- [ ] Update 阶段无依赖关系顺序问题
- [ ] 无无原因的 TBD/TODO 占位符
- [ ] 开放问题节中无技术选型或算法遗留（只含用户显式标注为「后续版本」的项目）

所有条目通过后，再派发审查子智能体。

### 第 7 步：派发审查子智能体

使用 `spec-reviewer-prompt.md` 中的模板，派发审查子智能体（标准模型），提供：
- spec 文件路径
- 该板块在 modules.md 中的定义段落（协调者提取后传入）

审查结果：
- Approved → 提交该 spec 文件到 git，继续下一个板块
- Issues Found → 修改 spec → 重新派发同一审查子智能体重审（不跳过重审）

## 所有板块完成后

列出所有生成的 spec 文件：

```
所有板块的 spec 已生成完毕：

- docs/my-powers-output/specs/YYYY-MM-DD-<module-1>-spec.md
- docs/my-powers-output/specs/YYYY-MM-DD-<module-2>-spec.md
- ...

请逐一审阅。对每个 spec，您可以：
- 确认无误
- 提出修改意见（我会修改后重新请您确认该 spec）
- 标注 leave it TODO（该板块暂不实现，保留文档）

所有 spec 确认后，我将进入下一阶段。
```

**等待用户对所有 spec 的明确确认。**

有修改请求时：修改对应 spec → 重新请用户确认该 spec → 继续等待其他 spec 的确认。

全部确认后：调用 splitting-specs 子技能 — 读取 `splitting-specs/SKILL.md` 技能文件。

## Red Flags

**绝不：**
- 在一条消息里问多个问题
- 跳过「迭代计划」章节
- 在用户未确认所有 spec 前调用 splitting-specs
- 审查子智能体未通过就提交 spec
- 审查发现问题后不重审直接提交
- Update 阶段有依赖关系但顺序未修正就通过审查
- 跳过初稿确认流程，直接编写正式 spec
- 所有要点已确认但未经用户同意就开始写正式 spec
- 把 modules.md 中涉及内部实现的描述当作已定结论，跳过对应讨论环节
- 以「待定」占位技术选型或算法——用户未显式说明「后续版本」时，必须在 spec 中做出选择，不遗留
