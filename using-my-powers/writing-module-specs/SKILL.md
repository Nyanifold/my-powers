# Writing Module Specs — 板块规格编写

对每个叶子板块依次进行头脑风暴，编写完整的规格文档。所有 spec 生成完毕后一并交用户审阅，用户确认后进入 splitting-specs。

**宣告：** 「我正在使用 my-powers:writing-module-specs，当前板块：<name>」

<HARD-GATE>
用户未确认所有 spec 之前，不得调用 splitting-specs。每个板块的 spec 必须包含「渐进式实现」章节。
</HARD-GATE>

## 处理顺序

1. 读取 `docs/YYYY-MM-DD-modules.md`，提取所有叶子板块列表
2. 按依赖顺序排列叶子板块（被依赖的板块先做）
3. 串行对每个板块执行「单板块头脑风暴流程」
4. 所有板块完成后，列出所有 spec 文件，请用户一并审阅
5. 用户确认（或标注 TODO）后，调用 splitting-specs

## 单板块头脑风暴流程

### 第 1 步：宣告与上下文回顾

宣告当前板块，并简要复述来自板块说明文档的定义：
> 「开始对板块「<name>」进行 spec 头脑风暴。根据板块说明文档，该板块的职责是 <...>，对外接口包括 <...>。」

### 第 2 步：逐一提问澄清需求

**规则：**
- 每次只问一个问题，等待用户回答后再问下一个
- 优先提供选项，减少用户思考负担
- 需要展示结构或关系时，可使用 ASCII 树、ASCII 框图或 mermaid/dot 语法辅助说明

**覆盖以下维度（根据板块性质选择适用的）：**

- **功能需求细化**
  - 核心功能的具体行为是什么？
  - 是否有优先级区分？
  
- **约束与非功能需求**
  - 性能要求？延迟上限？吞吐量？
  - 可靠性要求？是否需要重试、幂等？
  - 安全边界？哪些数据需要保护？

- **接口细节**
  - 数据结构的具体字段？
  - 错误情况如何分类和返回？
  - 是否有版本兼容性考虑？

- **与相邻板块的边界情况**
  - 依赖的板块返回错误时，本板块如何处理？
  - 是否有状态需要跨板块同步？

- **渐进式实现的划分**
  - 哪些功能是系统正常运行的最小集合？
  - 哪些功能可以后期叠加而不破坏接口？

### 第 3 步：呈现设计方案

分段呈现，每段确认后再继续：
1. **架构概述** — 板块内部的主要组成部分（复杂时附类图）
2. **接口细化** — 比板块说明文档更详细的接口定义（持久化板块附 ER 图）
3. **错误处理策略** — 各类错误的处理方式
4. **渐进式实现划分** — 基础功能 vs 增强功能的具体边界

### 第 4 步：编写 spec 文件

文件路径：`docs/specs/YYYY-MM-DD-<module>-spec.md`

```markdown
# <板块名> Spec

> 生成日期：YYYY-MM-DD
> 所属系统板块说明：docs/YYYY-MM-DD-modules.md

## 概述

<2-3 句话说明该板块的定位与作用>

## 职责与边界

**职责：**
<列表>

**不包含（边界）：**
<列表>

## 架构概述

<说明板块内部的主要组成部分及其协作方式>

<!-- 图表说明：
  - 内部结构简单（≤3 个类/组件）：文字描述或 ASCII 框图即可
  - 内部有多个类/结构体协作：附类图（UML Class Diagram），
    可用 mermaid/dot classDiagram 语法，展示类的属性、方法、继承与组合关系
  - 有复杂的并发流程或业务分支：附活动图（Activity Diagram），
    可用 mermaid/dot flowchart 语法
  - 板块是数据处理/ETL 类型：附数据流图（DFD），
    可用 mermaid/dot flowchart 或 ASCII 框图展示数据如何在内部流动和变换
-->

## 接口定义

### 对外暴露的接口

<每个接口一节，包括：>
- 名称与协议（HTTP/gRPC/消息队列/函数调用等）
- 请求/输入参数（字段名、类型、是否必填、说明）
- 响应/输出（字段名、类型、说明）
- 错误码与含义

<!-- 图表说明（持久化板块）：
  如果该板块管理持久化数据，在此附 ER 图，
  展示实体、字段类型、主键/外键、关系（一对多/多对多等）。
  可用 mermaid/dot erDiagram 语法。
  ER 图是接口定义的一部分，不是可选附录。
-->

### 依赖的外部接口

<本板块调用其他板块的接口，同样格式>

## 功能需求

<每个功能一节，描述预期行为，包括正常路径和异常路径>

<!-- 图表说明：
  - 功能涉及明确的状态转换（订单状态、连接状态、任务生命周期等）：
    附状态机图（State Machine Diagram），展示状态、触发事件、转换条件。
    可用 mermaid/dot stateDiagram-v2 语法。
  - 判断标准：如果一个对象有超过 3 种状态且转换条件不同，就值得画状态机图
-->

## 非功能需求

- **性能：** <延迟/吞吐量要求，或"暂无硬性要求">
- **可靠性：** <重试策略、幂等性要求>
- **安全：** <认证、授权、数据保护要求>

## 渐进式实现

### 基础功能

> 这些功能构成板块的最小可运行集合。基础版实现完成后，系统应能正常工作。

- <功能 1>：<简要说明>
- <功能 2>：<简要说明>

### 增强功能

> 这些功能在基础版之上叠加，不破坏现有接口。

- <增强功能 A>：<简要说明，包括与基础版的接口差异（如有）>
- <增强功能 B>：<简要说明>

## 开放问题（TODO）

<讨论中未确定的内容，标注 TODO，留给后续解决>
```

## 图表使用规范（spec 范围）

| 图表类型 | 放置章节 | 使用条件 |
|----------|----------|----------|
| 类图（UML Class Diagram） | 架构概述 | 板块内部有 ≥ 4 个类/结构体协作 |
| 活动图（Activity Diagram） | 架构概述 | 有复杂并发分支或多路径业务流程 |
| 数据流图（DFD） | 架构概述 | 数据处理/ETL 类板块，强调数据变换而非控制流 |
| ER 图 | 接口定义 | 板块管理持久化数据（必须附，不可选） |
| 状态机图（State Machine） | 功能需求 | 存在 ≥ 3 种状态且转换条件不同的对象 |

**不在 spec 里放的图：** 时序图（跨板块交互）、组件图、部署图——这些属于 modules.md。

### 第 5 步：派发审查子智能体

使用 `spec-reviewer-prompt.md` 中的模板，派发审查子智能体（标准模型），提供：
- spec 文件路径
- 该板块在 modules.md 中的定义段落（协调者提取后传入）

审查结果：
- Approved → 提交该 spec 文件到 git，继续下一个板块
- Issues Found → 修改 spec → 重新派发同一审查子智能体重审（不跳过重审）

## 所有板块完成后

列出所有生成的 spec 文件：

```
所有板块的 spec 已生成完毕：

- docs/specs/YYYY-MM-DD-<module-1>-spec.md
- docs/specs/YYYY-MM-DD-<module-2>-spec.md
- ...

请逐一审阅。对每个 spec，您可以：
- 确认无误
- 提出修改意见（我会修改后重新请您确认该 spec）
- 标注 leave it TODO（该板块暂不实现，保留文档）

所有 spec 确认后，我将进入下一阶段。
```

**等待用户对所有 spec 的明确确认。**

有修改请求时：修改对应 spec → 重新请用户确认该 spec → 继续等待其他 spec 的确认。

全部确认后：调用 `my-powers:splitting-specs`。

## Red Flags

**绝不：**
- 在一条消息里问多个问题
- 跳过「渐进式实现」章节
- 在用户未确认所有 spec 前调用 splitting-specs
- 审查子智能体未通过就提交 spec
- 审查发现问题后不重审直接提交
- 增强功能有依赖关系但顺序未修正就通过审查
