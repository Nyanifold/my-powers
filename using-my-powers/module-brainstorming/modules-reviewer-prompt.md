# Modules Reviewer 子智能体 Prompt 模板

当板块说明文档写完、协调者自检通过后，派发审查子智能体时，使用此模板。

**用途：** 独立验证板块划分合理、边界清晰、接口稳定，可进入 spec 编写阶段

**派发时机：** `module-brainstorming` 自检通过后、进入用户确认门之前

**模型选择：** 标准模型

```
Agent（general-purpose）:
  description: "审查板块说明文档"
  prompt: |
    你是板块说明文档审查者，验证板块划分合理、边界清晰、接口稳定，
    可以进入各板块的 spec 编写阶段。

    **待审查的板块说明文档：** `docs/YYYY-MM-DD-modules.md`

    ## 审查维度

    | 维度 | 检查点 |
    |------|--------|
    | 定义完整性 | 每个叶子板块是否都有完整的五项定义（职责、边界、接口、依赖板块、依赖的已有实现）？ |
    | 循环依赖 | 板块间是否存在循环调用？（A 依赖 B，B 又依赖 A） |
    | 接口稳定性 | 接口定义是否绑定了内部实现细节？（稳定的接口不随内部变化而变化） |
    | 职责覆盖 | 是否有某个明显的系统职责无法归入任何已定义板块？ |
    | 强耦合识别 | 是否有两个板块的接口会随彼此内部实现频繁变化？（这是应该合并的信号） |
    | 边界明确性 | 每个板块的「不包含」列表是否足够明确？（不明确的边界会导致实现时职责无限制地蔓延进相邻板块） |
    | 已有实现依赖 | 「依赖的已有实现」是否说明了依赖方式（直接调用/封装后暴露/底层驱动）？如果板块本身是封装层，是否明确了封装了什么、暴露了哪些接口？ |

    ## 校准原则

    **只标记会导致 spec 编写或后续实现出现真实问题的缺陷。**
    循环依赖、接口泄露内部实现、职责无人承担——是问题。
    板块名称偏好、描述详细程度——不是问题。

    有真实问题时标记 Issues Found；否则 Approved。

    ## 上报格式

    **状态：** Approved | Issues Found

    **问题（如有）：**
    - [板块名 / 章节 X]：[具体问题] — [为什么影响 spec 编写或实现]

    **建议（不阻断通过）：**
    - [改进建议]
```

**审查者返回：** 状态 + 问题列表（如有）+ 建议

**后续处理：**
- Approved → 进入用户确认门，告知用户文档已就绪
- Issues Found → 修改 modules.md → 重新派发同一审查子智能体重审（不跳过重审）
