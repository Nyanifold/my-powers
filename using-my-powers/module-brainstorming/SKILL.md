# Module Brainstorming — 板块分解与边界定义

将系统分解为边界清晰、相对独立的板块。板块边界定义好之后，板块内部的修改不应影响其他板块对它的调用。

**宣告：** 「我正在使用 my-powers:module-brainstorming」

<HARD-GATE>
用户未确认板块说明文档之前，不得进入 writing-module-specs。板块边界模糊、接口未定义时，不得开始 spec 编写。
</HARD-GATE>

## 检查清单

按顺序完成以下每一项：

1. **探索现有上下文** — 如果是已有项目，检查代码结构、文档、近期提交
2. **讨论系统分解** — 逐一提问，识别系统的主要组成部分
3. **识别强耦合区** — 强耦合处不分解，保留为一个板块
4. **定义叶子板块** — 对每个最细粒度板块定义职责、边界、接口
5. **写板块说明文档** — 保存至 `docs/YYYY-MM-DD-modules.md`，提交 git
6. **等待用户确认** — 无异议后调用 writing-module-specs

## 讨论系统分解

**提问方式：**
- 每次只问一个问题
- 优先提供选项（服务端/客户端？前后端分离还是一体？）
- 从高层往细节走，不要一开始就问细节

**分解维度参考（根据系统类型选择适用的）：**
- 运行位置：服务端 / 客户端 / 边缘
- 职责层次：数据采集 / 处理 / 存储 / 展示
- 数据流向：生成 / 清洗 / 训练 / 推理 / 评估
- 部署单元：前端 / 后端 / 数据库 / 消息队列 / 工作进程
- 用户交互：API 层 / 业务逻辑层 / 持久层

**分解原则：**
- 分解是层次化的——先找顶层边界，再往下细化
- 强耦合处不要强行分解：如果两个部分的接口会随着内部实现频繁变化，说明它们应该是同一个板块
- 叶子板块的判断标准：给定边界与接口约定后，内部修改不影响外部调用
- **如果系统本身足够简单，整体就是一个板块，无需分解——这是合理结论，接受它；不要为了分解而分解**

**识别强耦合：**
- 询问：「这两个部分之间会共享哪些数据结构？」
- 询问：「如果 A 改了内部实现，B 需要跟着改吗？」
- 如果答案是"是"且无法通过接口隔离，则合并为一个板块

## 定义每个叶子板块

对每个叶子板块，明确以下内容：

**职责（做什么）：**
- 一句话说明这个板块的核心职责
- 列出它管理的主要数据或状态

**边界（不做什么）：**
- 明确列出不属于该板块的内容
- 这防止板块膨胀和职责扩散

**对外接口：**
- 暴露给其他板块的 API、消息格式、文件格式、协议
- 接口要足够稳定：不随内部实现变化而变化
- 此处只需定义接口形状（入参/出参/错误类型），不需要实现细节

**依赖的其他板块：**
- 列出调用方向（A 调用 B，而不是 B 调用 A）
- 如果有循环依赖，这是边界划分有问题的信号

## 图表使用规范（modules.md 范围）

| 图表类型 | 放置位置 | 使用条件 |
|----------|----------|----------|
| 组件图（Component Diagram） | 板块间依赖关系章节 | 板块 ≥ 6 个，或依赖关系复杂难以用文字说清 |
| 部署图（Deployment Diagram） | 板块间依赖关系章节 | 系统有多节点部署需求 |
| 时序图（Sequence Diagram） | 跨板块交互时序章节 | 用例涉及 ≥ 3 个板块的调用链，或含异步消息 |

**不在 modules.md 里放的图：** ER 图、类图、状态机图、活动图、数据流图——这些描述的是板块内部细节，属于各板块的 spec。

## 派发审查子智能体

自检通过后，使用 `modules-reviewer-prompt.md` 中的模板派发审查子智能体（标准模型），提供：
- 板块说明文档路径

审查结果：
- Approved → 进入用户确认门
- Issues Found → 修改 modules.md → 重新派发同一审查子智能体重审（不跳过重审）

## 用户确认门

文档写完后，告知用户：

> 「板块说明文档已写入 `docs/YYYY-MM-DD-modules.md` 并提交。请审阅板块划分与接口定义，确认无误后我将逐一开始每个板块的 spec 头脑风暴。如有调整请直接说明。」

等待用户明确确认。有任何修改请求，修改后重新确认，不得跳过。

确认后：调用 `my-powers:writing-module-specs`。

## Red Flags

**绝不：**
- 在边界模糊时开始 spec 编写
- 把接口定义得过于具体（绑定了内部实现）
- 强行分解强耦合部分
- 系统确实简单时还要强行分解（一个板块就是一个板块）
- 跳过用户确认直接进入下一阶段
- 在一条消息里问多个问题
- 审查子智能体未通过就进入用户确认门
- 审查发现问题后不重审直接继续
