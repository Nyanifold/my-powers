# Module Brainstorming — 板块分解与边界定义

将系统分解为边界清晰、相对独立的板块。板块边界定义好之后，板块内部的修改不应影响其他板块对它的调用。

**宣告：** 「我正在使用 my-powers:module-brainstorming」

**任务跟踪：** 使用 TaskCreate 为检查清单中的每一项创建任务，按顺序完成，完成后用 TaskUpdate 标记为已完成。

<HARD-GATE>
用户未确认板块说明文档之前，不得进入 writing-module-specs。板块边界模糊、接口未定义时，不得开始 spec 编写。
</HARD-GATE>

## Anti-Pattern：「这个系统不需要板块分解」

每个系统都走这个流程。"看起来简单"的系统最容易因为未检验的假设浪费后续时间。分解讨论可以很短——真正简单时结论就是「整体是一个板块」——但你必须走完讨论流程并得到用户确认，不能以"系统简单"为由跳过。

**「整体一个板块」是讨论后得出的合理结论，不是跳过讨论的理由。**

## 检查清单

按顺序完成以下每一项：

1. **探索现有上下文** — 如果是已有项目，检查代码结构、文档、近期提交
2. **讨论系统分解** — 逐一提问，识别系统的主要组成部分
3. **识别强耦合区** — 强耦合处不分解，保留为一个板块
4. **定义叶子板块** — 对每个最细粒度板块定义职责、边界、接口
5. **收集项目参考资料** — 检查当前目录是否有项目基础；用户是否提到参考资料。有则派发子智能体，按初步板块划分汇报各板块的参考文件或目录；无则记录"无参考资料"并继续
6. **写板块说明文档** — 先写 `docs/drafts/YYYY-MM-DD-modules-draft.md`，逐条确认要点，再写正式文档至 `docs/YYYY-MM-DD-modules.md`，提交 git
7. **等待用户确认** — 无异议后调用 writing-module-specs

## 讨论系统分解

**提问方式：**
- 每次只问一个问题
- 每次问题优先提供 3-5 个可选项，并为每个选项简要说明核心取舍，帮助用户做出有依据的选择（而不只是列出名称）。例：前后端分离——接口边界清晰、可独立迭代，但需要维护 API 契约；前后端一体——开发摩擦小、共享类型，但扩展时边界容易模糊
- 从高层往细节走，不要一开始就问细节

**分解维度参考（根据系统类型选择适用的）：**
- 运行位置：服务端 / 客户端 / 边缘
- 职责层次：数据采集 / 处理 / 存储 / 展示
- 数据流向：生成 / 清洗 / 训练 / 推理 / 评估
- 部署单元：前端 / 后端 / 数据库 / 消息队列 / 工作进程
- 用户交互：API 层 / 业务逻辑层 / 持久层

**分解原则：**
- 分解是层次化的——先找顶层边界，再往下细化
- 强耦合处不要强行分解：如果两个部分的接口会随着内部实现频繁变化，说明它们应该是同一个板块
- 叶子板块的判断标准：给定边界与接口约定后，内部修改不影响外部调用
- 叶子板块内部可以有若干子模块（内部组织细节，不单独写 spec 或任务文件）——这没有问题
- **如果系统本身足够简单，整体就是一个板块，无需分解——这是合理结论，接受它；不要为了分解而分解**

**识别强耦合：**
- 询问：「这两个部分之间会共享哪些数据结构？」
- 询问：「如果 A 改了内部实现，B 需要跟着改吗？」
- 如果答案是"是"且无法通过接口隔离，则合并为一个板块

## 定义每个叶子板块

对每个叶子板块，明确以下内容：

**职责（做什么）：**
- 一句话说明这个板块的核心职责
- 列出它管理的主要数据或状态

**边界（不做什么）：**
- 明确列出不属于该板块的内容
- 这防止板块膨胀和职责扩散

**对外接口：**
- 暴露给其他板块的 API、消息格式、文件格式、协议
- 接口要足够稳定：不随内部实现变化而变化
- 此处只需定义接口形状（入参/出参/错误类型），不需要实现细节

**依赖的其他板块：**
- 列出调用方向（A 调用 B，而不是 B 调用 A）
- 如果有循环依赖，这是边界划分有问题的信号

**依赖的已有实现：**
- 列出该板块会直接复用或封装的现有代码、库、服务或外部系统
- 说明依赖方式：直接调用 / 封装后暴露 / 作为底层驱动
- 如果该板块本身就是对已有实现的封装层，明确说明封装了什么、暴露了哪些接口
- 没有已有实现依赖时，标注「无」

## 编写草稿前：收集项目参考资料

在讨论完成、进入初稿编写之前，执行以下检查：

**检查条件（满足任一即触发）：**
- 当前工作目录下存在项目代码、配置文件、文档等基础内容
- 用户在对话中提到了参考资料（代码库、文档、设计稿等）

**有参考资料时：** 使用 `reference-collector-prompt.md` 中的模板派发 Explore 子智能体，提供：
- 初步板块划分结果（板块名称列表）
- 需要调查的目录或文件范围

子智能体完成后，将参考资料映射结果附入草稿，供后续 spec 阶段引用。如补充说明中有涉及板块划分的信息，先与用户讨论，视情况调整板块定义。

**无参考资料时：** 记录"当前无项目基础，无参考资料"，直接进入初稿确认流程。

## 图表使用规范（modules.md 范围）

| 图表类型 | 放置位置 | 使用条件 |
|----------|----------|----------|
| 组件图（Component Diagram） | 板块间依赖关系章节 | 板块 ≥ 6 个，或依赖关系复杂难以用文字说清 |
| 部署图（Deployment Diagram） | 板块间依赖关系章节 | 系统有多节点部署需求 |
| 时序图（Sequence Diagram） | 跨板块交互时序章节 | 用例涉及 ≥ 3 个板块的调用链，或含异步消息 |

**不在 modules.md 里放的图：** ER 图、类图、状态机图、活动图、数据流图——这些描述的是板块内部细节，属于各板块的 spec。

## 初稿确认流程

用户回答所有问题后，**在写任何正式文档之前**，执行以下步骤：

### 第 1 步：写初稿

将讨论结果写入 `docs/drafts/YYYY-MM-DD-modules-draft.md`，在文件中列出每个要点并标记 `- [ ]`：

```markdown
# 系统板块说明（初稿待确认）

## 要点确认清单

- [ ] **板块树结构**：[完整板块树]
- [ ] **<板块A> 定义**：职责：...；边界：...；接口：...
- [ ] **<板块B> 定义**：职责：...；边界：...；接口：...
- [ ] **板块间依赖**：...
```

每个叶子板块单独一条，包含职责、边界和接口摘要。

### 第 2 步：逐条确认

每次只发一条要点请用户确认：
> 「请确认板块树结构：[内容]，是否正确？」

用户确认后，将对应 `- [ ]` 改为 `- [x]`。  
用户有修改时，更新初稿并重新确认该条目，再继续下一条。

### 第 3 步：全部确认后请求开始

所有条目 `- [x]` 后，向用户确认：

> 「所有要点已确认。是否可以开始撰写正式板块说明文档 `docs/YYYY-MM-DD-modules.md`？」

等待用户明确同意后，按「板块说明文档结构」写正式文档，提交 git。

## 派发审查子智能体

自检通过后，使用 `modules-reviewer-prompt.md` 中的模板派发审查子智能体（标准模型），提供：
- 板块说明文档路径

审查结果：
- Approved → 进入用户确认门
- Issues Found → 修改 modules.md → 重新派发同一审查子智能体重审（不跳过重审）

## 用户确认门

文档写完后，告知用户：

> 「板块说明文档已写入 `docs/YYYY-MM-DD-modules.md` 并提交。请审阅板块划分与接口定义，确认无误后我将逐一开始每个板块的 spec 头脑风暴。如有调整请直接说明。」

等待用户明确确认。有任何修改请求，修改后重新确认，不得跳过。

确认后：调用 writing-module-specs 子技能 — 读取 `writing-module-specs/SKILL.md` 技能文件。

## Red Flags

**绝不：**
- 在边界模糊时开始 spec 编写
- 把接口定义得过于具体（绑定了内部实现）
- 强行分解强耦合部分
- 系统确实简单时还要强行分解（一个板块就是一个板块）
- 跳过用户确认直接进入下一阶段
- 在一条消息里问多个问题
- 审查子智能体未通过就进入用户确认门
- 审查发现问题后不重审直接继续
- 跳过初稿确认流程，直接写正式板块说明文档
- 有项目基础或用户提到参考资料时，跳过子智能体收集步骤直接写草稿
- 所有要点已确认但未经用户同意就开始写正式文档
