# Verification Before Completion — 验证先于声明

在没有验证的情况下声称工作已完成，是不诚实，而非高效。

**宣告：** 「我正在使用 my-powers:verification-before-completion」

**核心原则：** 证据先于声明，永远如此。

**违反本规则的字面内容，就是违反本规则的精神。**

## 铁律

```
没有当前会话中新鲜运行的验证证据，不得做任何完成声明
```

如果没有在本条消息中运行验证命令，不得声称它通过了。

## 验证门控

```
在声称任何状态或表达满意之前：

1. 确认：什么命令能证明这个声明？
2. 运行：完整执行该命令（新鲜、完整）
3. 阅读：完整输出，检查退出码，计数失败
4. 核实：输出是否确认了声明？
   - 如果否：用证据陈述实际状态
   - 如果是：附上证据陈述声明
5. 仅此之后：做出声明

跳过任何步骤 = 说谎，而非验证
```

## 常见失败

| 声明 | 需要 | 不够的 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 个失败 | 上次运行结果、"应该通过" |
| 代码无 lint 错误 | lint 输出：0 个错误 | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | lint 通过、日志看起来不错 |
| bug 已修复 | 测试原始症状：通过 | 代码已变更、假设已修复 |
| 回归测试有效 | 红绿循环已验证 | 测试通过一次 |
| 子智能体已完成 | VCS diff 显示变更 | 子智能体报告"成功" |
| 需求已满足 | 逐行核对清单 | 测试通过 |

## Red Flags — 停止

- 使用"应该"、"大概"、"看起来"
- 在验证之前表达满意（"太好了！"、"完美！"、"完成了！"等）
- 即将提交/推送/PR 但没有验证
- 相信子智能体的成功报告
- 依赖部分验证
- 认为"就这一次"
- 疲惫想要结束工作
- **任何暗示成功而没有运行验证的措辞**

## 防止借口

| 借口 | 现实 |
|------|------|
| "现在应该可以了" | **运行**验证命令 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "lint 通过了" | lint ≠ 编译器 |
| "子智能体说成功了" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查就够了" | 部分什么都证明不了 |
| "换个说法规则就不适用了" | 精神优先于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过了" / "看起来正确"
```

**回归测试（TDD 红绿）：**
```
✅ 写测试 → 运行（通过）→ 还原修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了一个回归测试"（没有验证红绿循环）
```

**构建：**
```
✅ [运行构建] [看到：exit 0] "构建通过"
❌ "lint 通过了"（lint 不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建核对清单 → 逐一验证 → 报告差距或完成
❌ "测试通过，阶段完成"
```

**子智能体委托：**
```
✅ 子智能体报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 相信子智能体报告
```

## 适用时机

**在以下情况之前始终适用：**
- 任何成功/完成声明的变体
- 任何满意的表达
- 任何关于工作状态的积极陈述
- 提交、创建 PR、标记任务完成
- 移到下一个任务
- 委托给子智能体

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后声明结果。

这是不可协商的。
