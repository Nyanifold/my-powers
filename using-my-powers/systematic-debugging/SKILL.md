# Systematic Debugging — 系统化调试

随机修复浪费时间，并会引入新的 bug。快速补丁会掩盖根本问题。

**宣告：** 「我正在使用 my-powers:systematic-debugging」

**核心原则：** 永远先找到根本原因，再尝试修复。修复症状是失败。

**违反本流程的字面规则，就是违反本流程的精神。**

## 铁律

```
没有完成根因调查，不得提出任何修复方案
```

未完成第一阶段，不得进入修复步骤。

## 适用场景

适用于所有技术问题：
- 测试失败
- 生产 bug
- 非预期行为
- 性能问题
- 构建失败
- 集成问题

**尤其要在以下情况使用：**
- 时间紧迫时（紧急情况更容易诱使猜测）
- "只需要一个快速修复"看起来很明显时
- 已经尝试了多次修复
- 上一次修复没有起效
- 没有完全理解问题时

**不要跳过的情况：**
- 问题看起来很简单（简单的 bug 也有根本原因）
- 时间紧迫（系统化调试比反复试错更快）

## 四个阶段

必须按顺序完成每个阶段，才能进入下一阶段。

### 第一阶段：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含精确的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能否可靠地触发？
   - 确切步骤是什么？
   - 是否每次都发生？
   - 如果无法复现 → 收集更多数据，不要猜测

3. **检查近期变更**
   - 什么变更可能导致了这个问题？
   - `git diff`、最近的提交
   - 新依赖、配置变更、环境差异

4. **多组件系统：添加诊断追踪**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复方案之前，先添加诊断追踪：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置的传播
     - 检查每层的状态

   运行一次以收集证据，确定在哪里出了问题
   然后分析证据，识别失败的组件
   然后调查该具体组件
   ```

5. **追踪数据流**

   **当错误深埋在调用栈中时：**

   参见本目录下的 `root-cause-tracing.md`。

   **简版：**
   - 错误值从哪里产生？
   - 什么代码传入了这个错误值？
   - 持续向上追踪，直到找到源头
   - 在源头修复，而不是在症状处修复

### 第二阶段：模式分析

**修复之前先找到模式：**

1. **找到可用的示例**
   - 在同一代码库中找到类似的可用代码
   - 有什么相似的东西是正常工作的？

2. **与参考对比**
   - 如果在实现某个模式，完整阅读参考实现
   - 不要略读——逐行阅读
   - 充分理解模式后再应用

3. **识别差异**
   - 可用的和不可用的之间有什么不同？
   - 列出每一个差异，无论多小
   - 不要假设"这个不重要"

4. **理解依赖关系**
   - 这个组件需要哪些其他组件？
   - 需要哪些设置、配置、环境？
   - 它做了哪些假设？

### 第三阶段：假设与验证

**科学方法：**

1. **形成单一假设**
   - 明确表述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做**最小可能的变更**来测试假设
   - 每次改变一个变量
   - 不要同时修复多个问题

3. **验证后再继续**
   - 有效了？是 → 第四阶段
   - 没效？形成**新假设**
   - **不要**在上面叠加更多修复

4. **不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 研究更多信息

### 第四阶段：实现

**修复根本原因，而不是症状：**

1. **创建失败的测试用例**
   - 尽可能简单的复现
   - 如果有测试框架则写自动化测试
   - 修复之前**必须**先有测试
   - 使用 `my-powers` 实现规范中的 TDD 流程编写正确的失败测试

2. **实现单一修复**
   - 解决已识别的根本原因
   - **一次一个变更**
   - 不要"顺手"改其他东西
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了？
   - 没有破坏其他测试？
   - 问题真正解决了？

4. **写 bugfix 记录**

   修复通过验证后，在 `docs/reports/` 写一份额外记录，文件名前缀与当前 stage 相同：

   ```
   docs/reports/YYYY-MM-DD-<module>-task-<N>-bugfix.md
   ```

   若同一 stage 内修复了多个 bug，按序编号：`-bugfix-1.md`、`-bugfix-2.md`。

   记录内容：
   ```markdown
   # <module> Stage <N> Bugfix 记录

   ## 症状
   <观察到的现象>

   ## 根本原因
   <根因调查结论>

   ## 修复方案
   <做了什么，为什么这样做>

   ## 验证
   <运行了什么命令，输出是什么>
   ```

   提交后继续原任务。

5. **修复无效时**
   - **停下来**
   - 计数：已经尝试了几次修复？
   - 少于 3 次：回到第一阶段，用新信息重新分析
   - **3 次或以上：停下来，质疑架构（见步骤 5）**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **3 次或以上修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每次修复都暴露不同位置的新共享状态/耦合/问题
   - 修复需要"大规模重构"才能实现
   - 每次修复都在其他地方产生新症状

   **停下来质疑基础问题：**
   - 这个模式从根本上是合理的吗？
   - 我们是否"出于惯性"坚持下去？
   - 是否应该重构架构，而不是继续修复症状？

   **在尝试更多修复之前，与用户讨论**

## Red Flags — 停止并遵循流程

如果发现自己在想：
- "先快速修复，之后再调查"
- "试着改一下 X 看看是否有效"
- "同时添加多个变更，运行测试"
- "跳过测试，手动验证就行了"
- "大概是 X，让我修复它"
- "没有完全理解，但这个可能有用"
- "已经在这里了，顺便改一改"
- "这些就是主要问题：[列出修复方案却没有调查]"
- 在追踪数据流之前提出解决方案
- **"再试一次修复"（已经试了 2 次以上）**
- **每次修复都在不同地方暴露新问题**

**以上所有情况意味着：停止。回到第一阶段。**

**如果 3 次以上修复失败：** 质疑架构（见第四阶段第 5 步）

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单的问题也有根本原因。流程对简单 bug 也很快。 |
| "紧急，没时间走流程" | 系统化调试比反复试错**更快**。 |
| "先试试这个，然后再调查" | 第一次修复设定了模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未经测试的修复无法持续。先写测试证明它有效。 |
| "一次多个修复节省时间" | 无法隔离哪个有效。会产生新 bug。 |
| "参考太长，我来改编一下" | 理解不完整必然产生 bug。完整阅读。 |
| "我看到问题了，让我修复" | 看到症状 ≠ 理解根本原因。 |
| "再试一次"（2 次失败后） | 3 次以上失败 = 架构问题。质疑模式，不要再修复了。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 读错误、复现、检查变更、收集证据 | 理解了什么和为什么 |
| **2. 模式** | 找可用示例、对比 | 识别出差异 |
| **3. 假设** | 形成理论、最小化测试 | 已确认或形成新假设 |
| **4. 实现** | 创建测试、修复、验证 | bug 解决，测试通过 |

## 当流程显示"没有根本原因"时

如果系统调查显示问题真的是环境性的、依赖时序的或外部的：

1. 流程已经完成
2. 记录你调查了什么
3. 实现适当的处理（重试、超时、错误信息）
4. 添加监控/日志以供未来调查

**但是：** 95% 的"没有根本原因"案例都是调查不完整。
